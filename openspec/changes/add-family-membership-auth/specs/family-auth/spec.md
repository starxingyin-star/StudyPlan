# Family Auth Spec

## ADDED Requirements

### Requirement: Caller identity is server-resolved

Cloud functions SHALL use the WeChat cloud runtime openid as the caller identity and SHALL NOT trust the client to choose a family id.

#### Scenario: Caller has no family

- **WHEN** a caller without active membership calls `bootstrapFamily`
- **THEN** the response includes `needsFamilySetup: true`
- **AND** no family data is returned as accessible data

### Requirement: Families are membership-scoped

Family data reads and writes SHALL use the caller's active membership family id.

#### Scenario: Caller belongs to a family

- **WHEN** a caller with active membership calls a family-scoped API
- **THEN** the handler reads and writes records for that membership's `familyId`

#### Scenario: Caller has no membership

- **WHEN** a caller without active membership calls a protected family-scoped API
- **THEN** the call is rejected before any family data write

### Requirement: Family invite joins

The system SHALL allow an existing family member to create a 6-character uppercase alphanumeric invite code and allow another WeChat caller to join with that code.

Invite codes SHALL be generated by the server, SHALL use an unambiguous 32-character alphabet, and SHALL NOT duplicate any other active, unexpired family invite code.

#### Scenario: Valid invite

- **WHEN** a caller joins with an active, unexpired invite code
- **THEN** the caller becomes an active member of the invite code's family

#### Scenario: Invite code collision

- **WHEN** a generated invite code already belongs to another active, unexpired family invite
- **THEN** the server generates another code before saving the invite

#### Scenario: Ambiguous invite code

- **WHEN** an invite code unexpectedly matches multiple active, unexpired families
- **THEN** joining with that code is rejected instead of choosing one family

### Requirement: Missing cloud collections are handled lazily

The system SHALL avoid full collection creation preflight on every API request and SHALL create only the collection needed by the current write when a collection is missing.

#### Scenario: Missing membership collection during read

- **WHEN** the `familyMembers` collection does not exist while resolving membership
- **THEN** the read behaves as an empty membership list instead of failing the API call

### Requirement: PIN is second-factor confirmation

The family PIN SHALL confirm sensitive in-family operations, but SHALL NOT grant family access by itself.

#### Scenario: Correct PIN from non-member

- **WHEN** a non-member submits the correct family PIN for a protected operation
- **THEN** the operation is rejected because membership is missing
